\textbf{Reviewer: 1}

\textbf{Major comments:}\\
\textit{\textbf{1. Generalisability beyond PID use cases.}\\
While the authors claim broad applicability (e.g., “generalizable to genetic diagnosis across disease areas”), all benchmarking is exclusively from primary immunodeficiency (PID) studies. Although the authors state that primary immunodeficiency (PID) constitutes their primary focus, there is no quantitative or qualitative demonstration that PanelAppRex works equally well for other disease categories, particularly those with more complex gene-phenotype relationships (e.g., neurogenetic disorders, cancer panels). }

%\textbf{Response:} Thank you. We agree and have therefore expanded the validation studies from five to fifteen. We include the specific focus on neurology, and as you suggested the complexity does appear more prominently and the related panels are notably larger. We also added a third cross-disciplinary category to provide a broader sample. 
%
%Under the recommended computational use, where all returned panels are considered jointly, the main result remained unchanged: in all fifteen studies, the query results included the causal gene. For manual use, the causal gene was present in 85.6\% of all returned panels, and it was present in the subjective user-selected best panel in all fifteen studies. The increased sample size provides a more informative benchmark while remaining readable and reproducible.


\textbf{Response:} Thank you. We agree and have expanded the benchmark from five to fifteen case studies, comprising five immunology, five neurology, and five cross-disciplinary cases. This broader set is now described in the Methods and Results and reported in Figure 1 and Table S1.

Using the recommended computational interpretation, in which all returned panels are considered jointly, at least one returned panel contained the causal gene in all fifteen cases. Across all returned panels, the causal gene was present in 85.6\% of panels, and it was present in the user-selected best-fit panel(s) in all fifteen cases. We believe this provides a broader, while still manageable, benchmark across disease areas.

\textit{\textbf{2. Unclear integration of RAG (Retrieval-Augmented Generation).}\\
The manuscript mentions use of ``evidence-based RAG'' in multiple sections but does not explain how this is implemented. Since RAG is typically associated with LLM pipelines, readers need more detail on what data is being retrieved, how evidence is used to re-rank or refine query output, and whether generation plays any role.}

\textbf{Response:} Thank you. We agree that our previous wording was too broad and did not clearly distinguish the AI-based extension from the core search workflow. In the revised manuscript, we now clarify that the benchmarked PanelAppRex search operates on the structured core dataset using deterministic term-based matching, and that the AI-derived component was not used to re-rank or refine the reported query results.

Instead, we included the AI-derived layer as a demonstration of how the core resource can be extended. For this step, we retrieved mechanistic gene text from UniProtKB and combined it with PanelAppRex panel membership and metadata, including panel name, disease group, reported phenotypes, and mode of inheritance. We then used a fixed-prompt GPT-based model to generate concise panel-level summaries. These outputs are shown in the interface as an experimental ``Info'' field for each panel and are intended to provide rapid interpretive context rather than to determine panel ranking.

We have revised the Methods, Results, Discussion, and Supplemental accordingly so that the manuscript now states explicitly what data were retrieved, what was generated, and where the resulting summaries appear in the interface. We also clarify that the manual-query benchmarking reported here was performed on the core structured dataset and did not use the optional AI summary layer.

\textbf{Reviewer: 2}

\textbf{Major comments:}\\
\textit{\textbf{1. Performance comparison.}\\
When going through the initial steps of building PanelAppRex, the core of the tool can be understood to be providing credentialed access to the APIs of existing resources, such as GE’s PanelApp. It would be helpful to understand whether further steps taken to make the tool robust and comprehensive, such as using RAG or incorporating additional features, provide an improvement in performance through a comparative analysis, perhaps using the same benchmarking framework, against simply querying the other APIs.}

\textbf{Response:} Thank you. We agree that comparison with an upstream API is a useful baseline. We therefore added a supplementary benchmark comparing PanelAppRex with a GE PanelApp API workflow using the same case-study query, ``\textit{SERPING1} Factor XII and edema''. This analysis is now described in the Supplementary Methods and shown in Figure S4.

Because the GE PanelApp API does not accept the same phenotype-style query and return ranked panel results directly, this comparison is necessarily indirect. The API workflow required repeated panel-level retrieval and local assembly before ranking, whereas PanelAppRex queried a pre-built local resource. We therefore present this analysis as a practical supplementary comparison rather than a one-to-one benchmark.

In this benchmark, PanelAppRex was faster at each tested subset size for time to final ranked result. The API workflow was also constrained by rate limiting, with HTTP 429 errors occurring at higher numbers of sequential panel queries. We also note in the revised manuscript that live primary-source APIs provide more up-to-date content than an offline snapshot, which remains an important advantage of those resources.
