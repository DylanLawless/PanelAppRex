\textbf{Reviewer: 1}

\textbf{Major comments:}\\
\textit{\textbf{1. Generalisability beyond PID use cases.}\\
While the authors claim broad applicability (e.g., “generalizable to genetic diagnosis across disease areas”), all benchmarking is exclusively from primary immunodeficiency (PID) studies. Although the authors state that primary immunodeficiency (PID) constitutes their primary focus, there is no quantitative or qualitative demonstration that PanelAppRex works equally well for other disease categories, particularly those with more complex gene-phenotype relationships (e.g., neurogenetic disorders, cancer panels). }

\textbf{Response:} Thank you. We agree and have thus expanded the validation studies from five to fifteen. We include the specific focus on neurology, and as you suggested the complexity does appear more prominently and the related panels are notably larger. We added a third category of cross-disciple to get a random sample. The main result, asking whether the causal gene being present in the query output, remained unchanged at 100\% in each study. The increased sample size provides more accurate measurements, of course. We believe this increased sample size provides a fair and reproducible benchmark while still being readable and without adding overhead noise or summary statistics. 

\textit{\textbf{2. Unclear integration of RAG (Retrieval-Augmented Generation).}\\
The manuscript mentions use of ``evidence-based RAG'' in multiple sections but does not explain how this is implemented. Since RAG is typically associated with LLM pipelines, readers need more detail on what data is being retrieved, how evidence is used to re-rank or refine query output, and whether generation plays any role.}
 
\textbf{Response:} Thank you. We originally considered whether or not to include RAG data in he main results and therefore failed to report it thoroughly. We have now updated to include the full protocol in the open-source repository, and include formal sections in both the methods and results to demonstrate how PanelAppRex's core dataset can be extended and interpreted using this method. We complete the practical demonstration by include the resulting user information button (``Info'') for each panel.  The adds a highly condensed, structured summary which is specific to each unique panel.

\textbf{Reviewer: 2}

\textbf{Major comments:}\\
\textit{\textbf{1. Performance comparison.}\\
When going through the initial steps of building PanelAppRex, the core of the tool can be understood to be providing credentialed access to the APIs of existing resources, such as GE’s PanelApp. It would be helpful to understand whether further steps taken to make the tool robust and comprehensive, such as using RAG or incorporating additional features, provide an improvement in performance through a comparative analysis, perhaps using the same benchmarking framework, against simply querying the other APIs.}

\textbf{Response:} Thank you.
We have update the manuscript to reflect the claims and practical constraint that motivates our design: PanelAppRex can be downloaded and run locally in protected environments. To demonstrate the accuracy of recovering underlining data, we have clarified the reporting in Figure S1. 

We were also interested in running a direct comparison against “querying the APIs”, because it is an intuitive baseline. In practice, we found two issues that could make such a comparison misleading. First, PanelAppRex is designed to run locally in protected clinical and HPC environments where  internet access is commonly restricted, where live API access is not possible in our intended use setting.
Second, the ``equivalent'' dataset used by PanelAppRex is assembled from multiple sources that are not currently available from any single API. Building an API-based comparator would therefore require us to design and implement a 
parallel 
%multi-source aggregation and harmonisation pipeline for the 
control arm -- effectively recreating the contribution and making the comparison unfair. 

Based on your recommendation, we have added this rationale in the manuscript. We also noted the limitation of an offline database compared to the strengths of primary-source APIs: for rapidly changing resources, direct API access can provide more frequent updates than an offline snapshot, which we cannot match in the same way.
